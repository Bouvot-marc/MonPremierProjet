Cours Git — Guide rapide

Introduction aux concepts et commandes essentielles de Git pour débutants et rappel rapide.

Qu'est‑ce que Git ?

Git est un système de gestion de versions distribué. Il permet de suivre l'historique des fichiers, collaborer avec d'autres développeurs et gérer différentes lignes de développement (branches).

  • Localement : historique, commits, branches.
  • À distance : partage via git remote / git push / git pull.

Installation

  • Windows : installer Git for Windows (inclut Git Bash).
  • macOS : via Homebrew brew install git ou Xcode tools.
  • Linux : sudo apt install git (Debian/Ubuntu) ou le gestionnaire de paquets de la distribution.

Configurer votre identité :

$ git config --global user.name "Prénom Nom"
$ git config --global user.email "moi@example.com"

Flux de travail de base

  1. Créer ou cloner un dépôt.
  2. Modifier des fichiers.
  3. Ajouter à l'index (staging area).
  4. Créer un commit.
  5. Envoyer vers le dépôt distant.

Exemples courants

# Créer un dépôt
$ git init

# Cloner un dépôt distant
$ git clone git@github.com:utilisateur/projet.git

# Vérifier l'état
$ git status

# Ajouter et committer
$ git add fichier.txt
$ git commit -m "Message clair décrivant le changement"

# Voir l'historique
$ git log --oneline --graph --decorate --all

Branches

Les branches isolent les lignes de développement.

# Créer et changer de branche
$ git branch ma-feature
$ git checkout ma-feature
# ou en une seule commande
$ git switch -c ma-feature

# Fusionner la branche dans main
$ git checkout main
$ git merge ma-feature

# Supprimer une branche locale
$ git branch -d ma-feature

Conseil : travaillez sur des branches courtes et faites des PR (pull requests) pour la revue.

Dépôts distants

# Ajouter un remote
$ git remote add origin git@github.com:utilisateur/projet.git

# Pousser la branche locale vers le remote
$ git push -u origin main

# Récupérer les changements et fusionner (pull)
$ git pull origin main

Utilisez git fetch + git merge ou git pull --rebase pour un historique linéaire si souhaité.

Résolution de conflits

Si deux commits modifient la même partie d'un fichier, Git crée un conflit. Éditer, tester, puis :

$ git add fichier-conflit
$ git commit

Annuler des changements

# Retirer du staging
$ git restore --staged fichier

# Annuler les changements locaux dans un fichier
$ git restore fichier

# Revenir au commit précédent (attention : destructive)
$ git reset --hard HEAD~1

Bonnes pratiques

  • Commits petits et atomiques avec des messages clairs.
  • Utiliser des branches pour les nouvelles fonctionnalités et corrections.
  • Faire des revues de code via PR/MR avant merge.
  • Pull régulièrement pour rester synchronisé avec l'équipe.
  • Ne pas committer de fichiers sensibles (utiliser .gitignore).
# Exemple .gitignore minimal
node_modules/
.DS_Store
.env

Cheat sheet rapide

git init                # init dépôt
git clone           # cloner
git status               # état
git add               # stage
git commit -m "msg"      # commit
git log --oneline        # historique court
git branch               # lister branches
git checkout          # changer de branche
git switch -c         # créer+switch
git merge             # fusionner
git fetch && git merge   # récupérer sans auto-pull
git pull                 # fetch+merge
git push origin       # envoyer
